## Context

当前 `mcp-gateway-web` 已支持多来源接口定义导入，并复用统一的导入确认与保存流程。现状中 Swagger JSON 在部分版本（2.9.2）导入失败，而用户预期其行为应与既有 Swagger/OpenAPI 导入体验一致。该修复需在现有架构内完成，避免对已稳定的 OpenAPI 导入链路产生行为变化。

## Goals / Non-Goals

**Goals:**
- 让 Swagger 2.9.2 JSON 可被正确识别与解析，并进入现有导入流程。
- 明确 Swagger 2.x 与 OpenAPI 的输入分流边界，避免兼容逻辑误作用到 OpenAPI。
- 增加覆盖 Swagger 2.9.2 成功导入与 OpenAPI 不回归的测试用例。

**Non-Goals:**
- 不新增导入来源类型。
- 不重构整个导入架构或改造预览/确认交互。
- 不引入新三方依赖或更改对外 API 协议。

## Decisions

- 决策 1：在来源解析入口增加轻量版本识别分支，先判定 Swagger 2.x 与 OpenAPI，再进入对应解析器。
  - 原因：以最小变更隔离兼容逻辑，降低对现有 OpenAPI 解析路径的影响。
  - 备选方案：在统一解析器内追加大量条件分支。未采用原因是耦合度更高、回归风险更大。

- 决策 2：针对 Swagger 2.9.2 的失败点仅补齐必要字段容错与结构兼容，不改变标准化输出模型。
  - 原因：保持后续预览、确认、入库链路稳定，减少联动改动。
  - 备选方案：扩展输出模型承载来源特有字段。未采用原因是会引入额外行为与维护成本。

- 决策 3：回归保障以“同一输入在 OpenAPI 路径输出不变”为验收基线。
  - 原因：问题修复目标明确要求不影响既有 OpenAPI 导入能力。
  - 备选方案：仅验证 Swagger 用例。未采用原因是无法证明无回归。

## Risks / Trade-offs

- [版本识别误判] Swagger/OpenAPI 边界条件处理不当可能导致走错解析器 → 通过添加版本识别测试与典型样例覆盖降低风险。
- [隐性字段差异] Swagger 2.9.2 中可选字段缺失可能再次触发解析异常 → 在关键读取点增加容错默认值并补充失败路径测试。
- [维护成本] 兼容逻辑分支增加后续维护复杂度 → 将分支集中在入口层并保持解析器职责单一，减少扩散。
