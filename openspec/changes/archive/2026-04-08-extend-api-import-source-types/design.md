## Context

当前导入页面已提供多种入口卡片，但实际只有手动导入与 OpenAPI/Swagger 导入完成端到端能力。现状问题是：来源格式与解析器未统一，导致新增来源时容易重复开发、交互不一致、字段映射规则分散。约束条件包括：
- 必须复用现有接口规范与保存链路，避免改变既有数据模型。
- 新增类型的导入体验需与 Swagger 流程保持一致（导入、解析、预览、确认、入库）。
- 变更范围最小化，不引入额外依赖与后端接口改造。

相关角色：产品/前端开发/使用方（接口维护人员）。

## Goals / Non-Goals

**Goals:**
- 补齐现有入口中未实现的 `Postman`、`cURL`、`Apifox` 导入能力，并在本次迭代新增 `HAR` 导入能力。
- 抽象统一的“来源解析 -> 标准化模型”转换层，复用 Swagger 已验证的后续流程。
- 形成可执行的来源选型标准（使用频率、社区成熟度、文档质量），指导下一轮扩展优先级。

**Non-Goals:**
- 不改造接口保存协议与后端数据结构。
- 不在本次引入低频或文档不完整的长尾来源。
- 不改变手动导入与 Swagger 的现有交互路径与行为语义。

## Decisions

1. 统一导入流水线，新增来源只负责“前置解析”
- 决策：将所有来源先转换为系统统一中间结构，再进入现有 Swagger 导入后的预览与保存流程。
- 原因：最大化复用已稳定流程，降低新增来源的行为偏差。
- 备选方案：为每种来源各自实现完整导入流程。未采纳原因是维护成本高且一致性差。

2. 采用“来源适配器”模式组织解析逻辑
- 决策：按来源拆分解析适配器（Postman/cURL/Apifox/HAR），由统一调度层调用并返回标准化结果。
- 原因：满足单一职责，便于独立测试和后续扩展。
- 备选方案：在页面组件内堆叠条件分支直接解析。未采纳原因是耦合过高，难以维护。

5. 解析结果展示与导入确认抽取为公用组件
- 决策：将“解析后树形展示、搜索、勾选、确认导入”抽成公用 Vue 组件，各来源解析页面复用。
- 原因：减少重复 UI 与交互逻辑，新增来源时只关注“输入 + 解析”。
- 备选方案：每种来源保留独立展示模板。未采纳原因是维护成本高且易出现交互不一致。

6. 解析后的统一结构单独类型文件定义
- 决策：定义单独 TypeScript 类型文件，统一描述“解析结果”结构（方法、路径、描述、可选标准化 ApiData 等）。
- 原因：保证 OpenAPI 与 Postman/cURL/Apifox/HAR 在解析结果层可对齐，降低跨组件适配成本。
- 备选方案：各页面内局部定义。未采纳原因是类型漂移风险高。

3. 优先支持高频且规范稳定的来源
- 决策：按“使用频率 + 社区成熟度 + 文档完整度”打分，本次范围确定为 Postman、cURL、Apifox，并新增 HAR；其他候选来源延后。
- 原因：确保功能投入产出比与可持续维护性。
- 备选方案：一次性扩展更多来源（如 HAR/Insomnia）。未采纳原因是需求不确定且测试成本过高。

4. 保持与 Swagger 导入一致的用户流程
- 决策：新增来源入口仍走与 Swagger 相同的步骤与确认机制，避免新增学习成本。
- 原因：降低行为差异导致的操作错误，减少培训成本。
- 备选方案：为不同来源做差异化流程。未采纳原因是会增加认知负担。

## Risks / Trade-offs

- [不同来源字段语义不一致导致映射丢失] -> 在适配器层提供显式映射表与缺失字段兜底策略，并在预览阶段标识降级项。
- [cURL 文本变体复杂，解析失败率偏高] -> 首版限定支持常见语法子集并给出可读错误提示，逐步扩展解析覆盖率。
- [Apifox 导出格式版本差异] -> 以主流稳定版本文档为基线，增加版本检测与兼容分支。
- [HAR 中存在噪声请求（静态资源、预检等）] -> 增加可导入请求过滤规则，仅保留符合接口规范的请求记录。
- [过度追求通用化导致实现复杂] -> 仅抽象最小必要接口，避免提前设计。
